Apache Struts 1

  • Push/Pull：Push
  • 难易：相对简单
  • 准确描述Struts1的词将会是，“永垂不朽”。十多年前从它开始学习，昨天还在用。是的，昨天！
  • 其他：模板用Tiles

Apache Click

  • Push/Pull：Pull
  • 难易：相对简单
  • 恶：组件需要用Java代码生成。
  • 其他：模板也是需要类继承。

Apache Wicket

  • Push/Pull：Pull
  • 难易：相对复杂
  • 恶：自由的虽然高，但是过于复杂。
  • 好：有些情况下可以改变组件的HTML路径——IResourceFinder；对设计人员也会比较亲切——得益于wicket:id的使用。(用CSS的Selector我也很喜欢，但是貌似增加了不必要的难度。)
  • 其他：模板也是需要类继承。

JSF2

  • Push/Pull：Push
  • 难易：中等
  • 恶：比如PRG的实现。
  • 好：jsfc应该不错，方便设计人员，即便对开发人员也应该有帮助。
  • 其他：忘了JSF1吧。

Seasar2

Seasar是和Spring，Guice相当的AOP/DI框架。同时对表现层做了些整合，包括Struts1，JSF1和Wicket(截至2011年底仍然Sandbox)等。

官方推荐这两个：

  • SAStruts：Strut1。整合得相当到位，发挥了Struts1的极致
  • Teeda：JSF1的实现。直接使用HTML的ID属性的想法不错。

WebObjects

  • Push/Pull：Pull
  • 难易：中等
  • 恶：同Wicket等控件模式的一样，绑定得很紧。
  • 其他：当初用的是4.x来着，从开发工具到Application服务器全部集成在一起。真不愧是Apple的大一统风范！

GWT

  • Push/Pull：Pull
  • 难易：中等
  • 恶：开发人员用Java，生成代码几乎全部是Javascript，设计人员如何插手？
  • 好：设计人员如果有GWT知识背景估计会很Happy。或者队伍成员只会Java......
  • 好：控件丰富。

个人理想中的

我认为，随着HTML5的推进，这层可以直接扔掉！好比是“反GWT”！是的！把整个服务器端的view层都扔掉！即使是一个轮子，能砍去也好！！！
